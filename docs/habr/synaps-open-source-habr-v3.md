# Планирование без чёрного ящика: что реально умеет SynAPS и почему 50K пока не взлетел

31 марта 2026 года я попал на открытую лекцию MOSITLAB про ИИ на производстве. На слайдах был кейс для Москабельмета: десятки тысяч операций, сотни рабочих центров, огромная матрица переналадок, заметная экономия. В центре была нейросеть.

У меня на таких кейсах один вопрос: почему система приняла именно это решение?

Если ответ сводится к "модель так решила", в производственном планировании это плохой фундамент. Ошибка тут стоит дорого, а отладка "по ощущениям" обычно не работает.

Поэтому я делаю [SynAPS](https://github.com/KonkovDV/SynAPS) как противоположный полюс: APS-движок без нейросетевого чёрного ящика, где можно выбрать конкретную конфигурацию решателя, зафиксировать параметры (включая `random_seed`), прогнать независимую проверку допустимости и получить артефакт прогона.

В репозитории уже есть воспроизводимый 50K прогон. Он полезен именно тем, что не рисует победу там, где её пока нет: на синтетическом инстансе 50 000 операций текущий `RHC` не успевает собрать допустимое полное расписание в таймбоксе этого исследования. И это зафиксировано JSON артефактом, который лежит рядом с кодом.

## О каком классе задач речь

Формальная вывеска длинная: MO-FJSP-SDST-ARC. По сути это про то, что нужно одновременно:

- разложить операции по рабочим центрам (с учетом допустимых машин)
- уважать порядок операций внутри заказа (предшествование)
- учитывать переналадки, зависящие от последовательности (SDST), и связанные потери
- не превышать доступность вспомогательных ресурсов (оснастка, бригада, инструмент)
- оптимизировать несколько конфликтующих целей (длина горизонта / makespan, переналадки, просрочки / tardiness, потери материала)

Это NP-трудная история. Один "точный решатель на все" быстро упирается в стену, а одна жадная эвристика дает красивый тайминг, но часто слабое качество.

## Что такое SynAPS сегодня

SynAPS - это не один решатель, а портфель с явными конфигурациями.

Часть конфигураций стохастические, но `random_seed` и ключевые параметры управляемы (в том числе через контракты solve/repair).

Отдельно про CP-SAT: в OR-Tools многопоточность (`num_workers > 1`) не гарантирует бит-в-бит одинаковый результат между запусками и машинами. Если это важно, фиксируйте `random_seed` и запускайте CP-SAT в одном потоке (`num_workers = 1`).

| Слой | Конфигурации / компоненты | Зачем |
|---|---|---|
| Точный | `CPSAT-*`, `CPSAT-EPS-*`, `CPSAT-PARETO-*` | малые и средние инстансы, где важны нижние границы и контролируемый разрыв (`gap`) |
| Декомпозиция | `LBBD-*`, `LBBD-*-HD` | когда монолитный CP-SAT уже не тянет размер |
| Конструктивный | `GREED*`, `BEAM-*` | быстрый детерминированный план "с нуля" |
| Крупный масштаб | `ALNS-*`, `RHC-*` | поиск узких мест на больших синтетических инстансах |
| Ремонт | `IncrementalRepair` + контракт ремонта (repair) | локальная починка уже существующего плана после сбоя |
| Контроль качества | `FeasibilityChecker` | независимая проверка допустимости результата |

Публичный реестр портфеля сейчас содержит 22 именованные конфигурации. Авторитетный список лежит в [synaps/solvers/registry.py][registry].

Если хочется быстро посмотреть, что считается публичным интерфейсом портфеля, точка входа - `available_solver_configs()` в том же файле.

Еще две вещи, которые я считаю частью "белого ящика":

- стабильные JSON контракты для solve/repair (Pydantic + JSON Schema): [synaps/contracts.py][contracts]
- отдельная проверка допустимости результата: [synaps/solvers/feasibility_checker.py][feasibility] (в текущей валидации она запускается для `status=feasible/optimal`; для `status=error` результат считается недопустимым без списка нарушений)

## Что изменилось в апреле 2026

Апрель был не про косметику.

1. `ALNS` и `RHC` окончательно стали частью публичного портфеля (их можно выбирать как конфигурации).
2. Появился отдельный воспроизводимый прогон для 50K сценария: [benchmark/study_rhc_50k.py][study].
3. `RHC` начал писать инженерные метрики в `solver_metadata` (например, `preprocessing_ms`, `peak_window_candidate_count`, `effective_window_operation_cap`, `time_limit_reached`).
4. На больших инстансах стало видно, что дорого обходится запасной ремонт хвоста после исчерпания бюджета времени. Поэтому при `time_limit_reached=true` хвостовая фаза ремонта пропускается (`fallback_repair_skipped=true`), а скрипт исследования сохраняет артефакт с частичным результатом и причиной остановки.

## 50K прогон: что говорит артефакт

Команда для локального повторения (лучше писать в отдельную папку, чтобы не перетирать артефакт в репозитории):

```bash
python -m benchmark.study_rhc_50k \
    --preset industrial-50k \
    --seeds 1 \
    --solvers RHC-GREEDY RHC-ALNS \
    --write-dir benchmark/studies/_local-rhc-50k
```

Артефакт, на который я ссылаюсь ниже, уже сохранен в репозитории: [benchmark/studies/2026-04-13-rhc-50k-machine-index/rhc_50k_study.json][artifact].

Этот файл не "презентационный". Он фиксирует ровно то, что реально получилось в заданном бюджете времени.

Мини-срез (для `RHC-GREEDY`):

```json
{
    "solver_config": "RHC-GREEDY",
    "results": {
        "status": "error",
        "feasible": false,
        "assignments": 6959
    },
    "statistics": {
        "wall_time_s_mean": 120.115
    },
    "solver_metadata": {
        "windows_solved": 11,
        "effective_window_operation_cap": 1040,
        "peak_window_candidate_count": 49931,
        "time_limit_reached": true,
        "fallback_repair_skipped": true
    }
}
```

Что это значит:

- `RHC-GREEDY` за 120.115 секунды успел зафиксировать 6 959 операций из 50 000 (11 окон)
- `RHC-ALNS` за 366.23 секунды успел зафиксировать 1 078 операций из 50 000 (3 окна)
- оба прогона завершились с `status=error`, то есть полное допустимое расписание не собрано
- в обоих случаях глобальный бюджет времени исчерпан (`time_limit_reached=true`), а "дочистка хвоста" пропущена (`fallback_repair_skipped=true`)
- `verification.violation_count=0` здесь не означает "нарушений нет": текущая `verify_schedule_result()` проверяет ограничения только если решатель заявил `status=feasible/optimal`; для `status=error` возвращается `feasible=false` без списка нарушений

Это неприятные цифры. Но это как раз то, что хочется видеть в инженерной системе: ошибка не маскируется, причины остановки видны.

## Где узкое место и почему оно уже не "первое окно"

Первая гипотеза была простой: RHC на первом окне затягивает почти весь инстанс и умирает.

В текущей реализации уже есть два "предохранителя": лимит операций на окно по оценке пропускной способности (`effective_window_operation_cap`) и индекс занятости машин (`MachineIndex`) для ускорения проверок слотов. Это помогает не взрывать внутреннюю сложность окна, но артефакт показывает, что кандидатный пул все равно остается почти полным. Значит "виновато только первое окно" уже не объясняет картину целиком.

При этом пик кандидатного пула почти равен всему инстансу (`peak_window_candidate_count` около 50k). То есть отбор операций в окно пока слишком широк: ранний фронтир (earliest-frontier) практически всегда "почти все".

На этом фоне `RHC-ALNS` особенно слаб: внутренний решатель ALNS не успевает сделать достаточно назначений, даже когда внешняя часть уже стала быстрее.

Отдельно видно еще одну дыру: `due_pressure_selected_ops` в текущем артефакте равно 0. То есть сигнал "дедлайн близко" пока почти не участвует в отборе операций (по крайней мере, на этом синтетическом инстансе).

Следующий шаг тут не в том, чтобы еще раз ускорить мелкие операции. Он про более агрессивное сужение кандидатного пула и про пропускную способность внутреннего решателя.

## Что дальше

Следующая практическая задача для SynAPS сейчас не звучит героически. Нужно научить `RHC` и его внутренний решатель работать быстрее на почти полном пуле раннего фронтира (earliest-frontier).

Если это получится, следующий 50K артефакт будет уже не только про частичный прогресс, а про заметно более сильное качество расписания. Если не получится, это тоже будет видно сразу.

В этом и смысл проекта: плановик получает несколько расписаний с числами, а не магию и не тысячу точек фронта Парето без контекста.

## Приложения

### Ремонт (IncrementalRepair)

В SynAPS есть отдельная поверхность для "починить уже существующий план", а не строить новый с нуля.

Вы передаете `base_assignments`, список "сломанных" операций (`disrupted_op_ids`) и радиус окрестности. Дальше движок:

1. фиксирует все назначения вне окрестности
2. пересобирает только локальный кусок (приоритетно по срочности заказа)
3. если конструктивный ремонт упирается в тупик, делает короткий запасной прогон CP-SAT (таймбокс 5 секунд)
4. пересчитывает `setup_minutes` по финальной последовательности, чтобы не появлялись "призрачные переналадки"

Код: [synaps/solvers/incremental_repair.py][incremental_repair], оркестрация: [synaps/portfolio.py][portfolio].

### Квантовый бэкенд (очень осторожно)

Иногда в планировании всплывает идея "закодируем в QUBO и отдадим квантовому отжигу". Формально можно написать что-то вроде:

$$H = A \sum_i \Bigl(1 - \sum_j x_{ij}\Bigr)^2 + B \sum_j \text{penalty}(j) + C \cdot \text{objective}$$

Практически на масштабах 50k операций это пока не про ближайшие годы. Поэтому в SynAPS это максимум потенциальная ветка в роутере в будущем, а не "готово к квантам" сегодня.
