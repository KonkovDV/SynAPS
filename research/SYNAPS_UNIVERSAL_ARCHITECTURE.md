# SynAPS: Универсальная платформа планирования — CURRENT vs TARGET

> **Классификация**: `CURRENT` = реализовано и верифицировано; `TARGET` = целевая архитектура.
> **Термины**: [GLOSSARY](../docs/partners/GLOSSARY_2026_04.md).

Система **SynAPS** — математическое и алгоритмическое ядро для решения NP-трудных задач теории расписаний. Архитектура спроектирована как отраслево-независимая: один solver-портфолио, конфигурируемый под домен через параметры, а не перезаписываемый под каждую вертикаль.

Формальный класс задач: $MO\text{-}FJSP\text{-}SDST\text{-}ARC$ (многокритериальное ресурсное планирование с зависящими от последовательности переналадками и ограничениями по вспомогательным ресурсам).

---

## 1. Универсальная Бизнес-Модель

Система устраняет «слепые зоны» человеческого управления за счёт разделения на два слоя:
* **Детерминированное ядро (Исполнитель)**: Обеспечивает строгую валидацию физической реальности. Гарантирует, что операция $A$ не начнётся раньше операции $B$, а ресурс не будет занят двумя задачами одновременно.
* **AI-советник (целевой слой)**: Оценивает комбинации и подсказывает лучший маршрут. **Не реализован в текущем репозитории** — описан как направление исследований.

### Точки монетизации:
1. **Минимизация переналадок**: Solver выстраивает задания так, чтобы ресурсы перенастраивались как можно реже. (`CURRENT` — CP-SAT solver с SDST-матрицами)
2. **Сглаживание энергопотребления**: Энергоёмкие процессы в целевой архитектуре должны сдвигаться в окна дешёвых тарифов. (`TARGET` — текущая схема уже хранит `energy_kwh` в setup entries, но активный evidence pack пока не фиксирует отдельный energy-cost objective в shipped solver portfolio)
3. **Хирургическая починка плана (Incremental Repair)**: При сбое ключевого узла расписание «лечится» только для затронутых компонентов. (`CURRENT` — `synaps/solvers/incremental_repair.py`)
4. **Управление дефицитными ресурсами (Auxiliary Resources)**: Оптимизация движения инструментов, тары, оснастки. (`CURRENT` — auxiliary-resource constraints в модели, тесты на feasibility)

## 2. Реализованная Техническая Архитектура (`CURRENT`)

### 2.1. Solver Portfolio

Каждый операционный контур имеет «бутылочное горлышко» (bottleneck), определяющее ритм всей системы.

| Solver | Роль | Файл | Статус |
| --- | --- | --- | --- |
| **Greedy Dispatch (ATCS)** | Быстрая начальная аллокация, log-space ATCS (Lee, Bhaskaran & Pinedo 1997) | `synaps/solvers/greedy_dispatch.py` | ✅ Реализован; active evidence pack фиксирует последний полностью задокументированный прогон `149/149`, при том что текущий репозиторий собирает `175` тестов |
| **CP-SAT** | Точный solver для bottleneck-подзадач (Google OR-Tools) | `synaps/solvers/cpsat_solver.py` | ✅ Реализован, SDST, multi-objective |
| **Incremental Repair** | Локальная починка при сбое | `synaps/solvers/incremental_repair.py` | ✅ Реализован |
| **LBBD** | Logic-Based Benders Decomposition — координация нескольких scheduling-слоёв | `synaps/solvers/lbbd_solver.py` | ✅ Реализован; публичная benchmark-оценка должна читаться через текущий evidence packet |
| **Pareto Slice** | Многокритериальная Парето-оптимизация | `synaps/solvers/pareto_slice_solver.py` | ✅ Реализован |
| **Portfolio Router** | Автоматический выбор solver-пути по профилю задачи | `synaps/solvers/router.py` | ✅ Реализован |

### 2.2. Универсальная Схема Данных

Слой бизнес-логики реализован через Universal SQL Schema (через поля `domain_attributes` в JSONB), что позволяет параметризировать систему для разных доменов:

* **Металлургия**: Тепловые режимы, прокатные станы
* **Фарма**: Ограничения на зачистку реакторов (CIP/SIP), окна годности
* **Электроника**: SMT-линии, матрицы смены кассет
* **FMCG**: Линии розлива, рецептурные переходы

Доменные примеры существуют в `schema/examples/`.

## 3. Целевая Архитектура (`TARGET`)

Следующие подсистемы описаны в архитектурных документах, но **не реализованы в текущем репозитории**.

### 3.1. Графовые Нейросети (HGAT) — `TARGET`

Проблема: агрегация тысяч операций в макро-сущности приводит к потере микро-деталей. Решение: моделирование операционного контура как графа через PyTorch Geometric, где связи инкапсулируют микроструктуру данных до агрегации.

### 3.2. Цифровой двойник и Offline RL — `TARGET`

Для обучения нейросетей нужна математическая песочница (SimPy / Rust DES). RL-агенты проходят миллионы симуляций, обучаясь восстанавливать расписание при сбоях (Conservative Q-Learning).

### 3.3. LLM Copilot On-Premises — `TARGET`

Локальная языковая модель (GLM-4/Qwen) для естественно-языкового интерфейса к ERP/MES. Оператор задаёт вопрос на русском или английском — система генерирует SQL-запрос и отвечает текстом. Данные не покидают периметр.

### 3.4. Межплощадочное Федеративное Обучение — `TARGET`

Flower-based FL: площадки обучаются коллективно без передачи коммерческих секретов, рецептур или данных о клиентах.

## Резюме

SynAPS — это **работающий Python-based solver portfolio** с универсальной схемой данных и benchmark-инфраструктурой. Принятие **гибридной стратегии** (точная математика + эвристики) делает планирование предсказуемым и воспроизводимым.

| Слой | Статус |
| --- | --- |
| Solver portfolio (6 путей) | ✅ `CURRENT` |
| Universal SQL Schema | ✅ `CURRENT` |
| Pydantic контракты + Feasibility | ✅ `CURRENT` |
| TypeScript BFF (минимальный) | ✅ `CURRENT` |
| Benchmark harness | ✅ `CURRENT` |
| HGAT / GNN advisory | ❌ `TARGET` |
| Digital Twin / RL | ❌ `TARGET` |
| LLM Copilot | ❌ `TARGET` |
| Federated Learning | ❌ `TARGET` |

AI-advisory, digital twin, LLM copilot и federated learning — это целевые слои, формирующие **roadmap**, а не текущий продукт.
