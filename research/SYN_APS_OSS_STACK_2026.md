# Syn-APS: Аудит и Обновление Open-Source Стека (Ревизия: Весна 2026)

Данный документ представляет собой срез самых свежих и актуальных Open-Source решений для архитектуры Syn-APS. Каждая подсистема проанализирована с точки зрения *State-of-the-Art (SOTA)* на 2026 год, чтобы гарантировать максимальную производительность, безопасность и долговечность (future-proofing) продукта.

---

## 1. Блок Оптимизации и Точной Математики (Solver Portfolio)

### В архитектуре: CP-SAT (Google) / HiGHS
* **Текущий статус (2026)**: В мире Open-Source точной математики (MIP/CP) **Google OR-Tools (версии 9.10+)** с их CP-SAT солвером остаётся абсолютным мировым лидером для задач типа *Job-Shop Scheduling*. 
* **Что касается LP/MIP**: **HiGHS (версии 1.8+)** закрепил за собой звание самого быстрого *открытого* линейного солвера, вытеснив старые Coin-OR (CBC).
* **Сценарий апгрейда**: 
  * *Вердикт*: Оставить связку OR-Tools (CP-SAT) + HiGHS. Это лучшие бесплатные решения, превосходящие многие коммерческие аналоги (кроме Gurobi).
  * *Свежий тренд*: Использовать интеграцию CP-SAT с языком **Rust** (через крейты-биндинги) для микросекундной генерации матриц, так как базовое ядро Syn-APS V3 планируется на Rust.

---

## 2. Блок Machine Learning: Графы и Оффлайн-Обучение

### 2.1. Графовые Нейросети (HGAT)
* **В архитектуре**: PyTorch Geometric (PyG).
* **Свежий тренд (2026)**: **PyTorch 2.6+** с механизмом `torch.compile` (Inductor) дает возможность компилировать графовые сети прямо в оптимизированный C++/Triton код. 
* **Сценарий апгрейда**: PyG остается стандартом де-факто для GNN. Однако для огромных фабрик (где графы не влезают в память одной GPU) стоит рассмотреть **DGL (Deep Graph Library)** от AWS, так как он лучше масштабируется на кластерах при *Distributed Graph Training*. Для Syn-APS (заводы средних размеров) оставляем **PyG + PyTorch 2.6+**.

### 2.2. Offline Reinforcement Learning (Политика агента)
* **В архитектуре**: Stable-Baselines3 (SB3) / d3rlpy.
* **Свежий тренд (2026)**: SB3 хорош для онлайн-обучения, но для чистых Offline-задач лидером стал **TorchRL** (официальная библиотека от Meta/PyTorch Core) и обновлённый **d3rlpy 3.x**.
* **Сценарий апгрейда**: Полный переход на **TorchRL**. Это даст нативную поддержку высокопроизводительных Replay Buffers и оптимизацию через Tensordict, что критически ускорит прогон через цифровой двойник.

---

## 3. Блок LLM и Natural Language (On-Premise Copilot)

### В архитектуре: vLLM + Llama / Qwen / GLM
* **Свежий тренд (2026)**:
  * **Inference Engine**: **SGLang** или **vLLM (v0.8+)**. SGLang (от LMSYS) показывает кардинально лучшую пропускную способность для задач со сложными промптами (RAG) благодаря архитектуре *RadixAttention*.
  * **Vector DB**: **pgvector** в составе **PostgreSQL 18** получил поддержку индексов *HNSW* на GPU.
  * **Модели**: Как разобрали ранее — **GLM-5.1** (для тяжелой инженерии), **Llama 3.1 8B** или **Qwen 2.5/3.0** (для легковесных агентов).
* **Сценарий апгрейда**: Разворачиваем инференс на базе **SGLang** (быстрее vLLM на 30% в длинных контекстах) + RAG на **PostgreSQL 18 (pgvector HNSW)**.

## 4. Блок Симуляции (Цифровой Двойник / Digital Twin)

### В архитектуре: SimPy (Python)
* **Свежий тренд (2026)**: SimPy невероятно удобен для Python-экосистемы, но он работает в один поток (ограничения GIL), что делает прогон миллионов сценариев Монте-Карло катастрофически медленным.
* **Сценарий апгрейда**: 
  * *Rust-симуляторы (например, ODES, Rust-DES)*. Так как ядро V3 планировалось на Rust (для детерминированности), имеет смысл переписать "математический движок симуляции" на Rust для параллелизма. 
  * Альтернатива в Python: **Salabim** (визуализация из коробки) или мультипроцессорная обертка над SimPy. Для *Offline RL* (где нужна генерация миллионов фреймов в час) **настоятельно рекомендуется Rust-backend** для симулятора.

---

## 5. Блок Edge AI (Микро-ИИ на оборудовании) и Federated Learning

### 5.1. Edge Computing
* **В архитектуре**: Apache TVM / TFLite Micro / ONNX.
* **Свежий тренд (2026)**: Meta выпустила **ExecuTorch** (официальный наследник PyTorch Mobile). Это революция для деплоя ИИ на микроконтроллеры (ПЛК, STM32).
* **Сценарий апгрейда**: Замена цепочек ONNX/TVM на **ExecuTorch**. Это позволяет экспортировать обученную графовую микро-модель напрямую из PyTorch на заводской контроллер без «костылей» промежуточных форматов.

### 5.2. Federated Learning (Мультизаводское обучение)
* **В архитектуре**: Flower (flwr).
* **Свежий тренд (2026)**: **Flower (v1.10+)** остается абсолютным монополистом в Open Source для FL. Платформа обросла мощными плагинами для Differential Privacy и Secure Aggregation.
* **Сценарий апгрейда**: Оставляем Flower, но включаем механизмы доверенного исполнения (*Confidential Computing*, например, Intel TDX / AMD SEV) на центральном сервере холдинга для криптографических гарантий приватности весов между заводами.

## 6. Базовый Бэкенд (Транспорт и Данные)

* **Очереди Ивентов**: **NATS JetStream (v2.11+)** — эталон для микросервисов IoT-класса (в 10 раз легче и быстрее Kafka на мелких сообщениях ПЛК-оборудования). Оставляем.
* **Оркестрация Workflow**: **Temporal (v1.26+)**. Незаменим для координации процессов, которые могут длиться часами (например, ожидание сборки сложного заказа). Оставляем.
* **Аналитика**: **ClickHouse** — безальтернативный SOTA для хранения телеметрии со станков и логов IoT (вставляет миллиарды строк в секунду).

---

### Итоговый SOTA-Стек Syn-APS 2026:
1. **Core Runtime**: Rust (performance) + Python 3.13 (ML).
2. **OR Engine**: Google OR-Tools (CP-SAT) + HiGHS.
3. **ML & GNN**: PyTorch 2.6 (TorchRL + Inductor) + PyTorch Geometric.
4. **LLM Copilot**: GLM-5.1 / Qwen 2.5 $\xrightarrow{}$ SGLang $\xrightarrow{}$ PostgreSQL 18 (pgvector).
5. **Edge AI**: ExecuTorch (запуск моделей на контроллерах станков).
6. **Data Transport**: NATS JetStream + Temporal.

Этот стек является **"выбором без компромиссов"** — он использует передовые открытые технологии, за которыми стоят тысячные комьюнити (Meta, Google, Z.ai), что полностью исключает vendor lock-in и обеспечивает промышленную надежность.
